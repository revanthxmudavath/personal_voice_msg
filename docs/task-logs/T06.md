## Scope and task split

- T06 owns provider-neutral search-result sanitization, a run-scoped private
  result-ID-to-URL map, and the secure `fetch_public_page(result_id)` transport.
- T07 owns real SearXNG request/parsing and Trafilatura extraction. Its trusted
  adapter will register parsed results into the T06 boundary.
- T09 owns any runtime discovery-agent benchmark. T06 gives deterministic or
  agentic callers the same narrow, enforceable web boundary.
- No agent-facing API accepts a URL. Full URLs remain private deterministic
  state; callers receive opaque random result IDs.
- Web metadata and fetched content remain untrusted. T06 does not extract,
  persist, approve, generate, synthesize, queue, or send content.

## Implemented boundary

- Added `aiohttp 3.14.2` as the smallest transport dependency and locked every
  transitive dependency.
- Search registration is atomic and limited to 20 results per run. Raw and
  sanitized title/snippet sizes are bounded, HTML is reduced to text, Unicode
  controls are removed, and all result IDs contain at least 192 bits of
  randomness.
- Result IDs are run-scoped and single-use. Fabricated, URL-shaped,
  cross-session, reused, closed-session, and concurrent requests fail with the
  same privacy-safe error.
- URL validation accepts only canonical public `http:80` and `https:443`
  destinations. It rejects credentials, fragments, non-default ports,
  ambiguous numeric forms, local suffixes, Unicode whitespace, controls, and
  non-public literal addresses.
- DNS resolution has a 3-second bound, rejects an answer set containing any
  non-public address, passes only validated numeric addresses to the connector,
  disables the connector DNS cache, revalidates the chosen socket address, and
  rejects a changed answer set during the fetch.
- Manual redirects enforce the same URL and DNS checks at every hop, reject
  loops and duplicates, permit at most five redirects, and reject HTTPS to HTTP
  downgrade.
- TLS uses the platform trust store while preserving the original hostname for
  HTTP Host, SNI, and certificate verification even though the validated
  numeric address is dialed.
- The client ignores ambient proxy variables, `.netrc`, cookies, and automatic
  decompression. It accepts only HTML, XHTML, or plain text and requires strict
  bounded headers and response framing.
- Bodies stream under a 1 MiB hard limit with a 15-second total deadline,
  3-second connect deadline, and 3-second idle-read deadline. A session permits
  only one in-flight fetch and cannot close during that operation.
- CI runs offline security tests normally and runs the real isolated network
  harness in a separate Docker job. Public live checks remain explicitly
  opt-in.

## Independent review

The first read-only security review found three blocking gaps: live tests were
selected by the normal security marker, controlled TLS identity had not been
proved, and several hostile framing/timeout cases lacked real endpoint tests.
It also identified raw metadata work, overlapping lifecycle operations, actual
`.netrc` coverage, and NAT64 scope as residual concerns.

The implementation and tests were tightened for every code-level finding:

- live tests moved to a separate live-only module;
- the Docker harness now proves trusted SNI/Host behavior and TLS failures;
- framing, downgrade, header, slow-stream, and premature-EOF cases were added;
- raw metadata is rejected before parsing;
- one in-flight operation is enforced;
- the runner contains a real `.netrc` and hostile proxy environment.

The final read-only re-review found no remaining production-code defect. It
confirmed that the earlier live-test, controlled TLS, hostile framing,
metadata-bound, concurrency, `.netrc`/proxy/cookie, and CI-separation findings
are closed. The reviewer independently reran locked sync, all 282 fast tests,
the 14 offline security tests, Ruff, strict mypy, repository policy, and Compose
validation. Docker execution was unavailable inside the reviewer's sandbox, so
the lead integrator's separate 30-case green Docker run is the authoritative
network-harness evidence.

## Explicit residual limits and deferrals

- Python blocks the standardized NAT64 well-known and local-use prefixes, but a
  deployment-specific Pref64 cannot be inferred portably from an individual
  address. T18 must prohibit NAT64 on discovery networks or supply and test the
  deployment Pref64 before release.
- An HTTP parser cannot treat bytes beyond a declared smaller Content-Length as
  part of that response. The client uses one request per forced-close connection
  so those bytes cannot be reused as another response; oversized declarations,
  premature EOF, transfer/content-length ambiguity, and streamed overflow are
  tested directly.
- DNS answer changes are tested directly against the same resolver instance and
  the production redirect path reuses that resolver. T18 still owns network
  policy and container egress enforcement.
- T07 must supply real SearXNG results through the trusted registration adapter;
  it must not add an arbitrary-URL path.
- Public live sites are useful compatibility checks but can change externally;
  only the deterministic Docker harness is a required CI gate.
