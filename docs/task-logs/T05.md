# T05 — Pacific-time scheduler and idempotent daily run

## Scope and assumptions

- T03 and T04 were complete before T05 began.
- Scheduling uses explicit timezone-aware inputs; no clock, timezone, or service
  clients are mocked.
- Daily preparation is scheduled for 06:50 and daily sending for 07:00 in
  `America/Los_Angeles`, as required by the implementation plan.
- The owner approved the explicit grace policy on 2026-07-21: weekly discovery
  is eligible Monday 00:00 through Tuesday 00:00 Pacific, preparation from
  06:50 through 07:00, and sending from 07:00 through 07:05. Start instants are
  inclusive and cutoff instants are exclusive.
- A missed preparation or send never catches up on the next Pacific date.
- A single scheduler service instance performs startup reconciliation. Durable
  uniqueness still protects against concurrent claims.
- T05 schedules and claims a daily run but does not implement content
  preparation, WhatsApp submission, retries, or delivery reconciliation; those
  remain in later tasks.

## Red evidence

- `tests/fast/test_scheduling.py` initially failed during collection with
  `ModuleNotFoundError: personal_voice_msg.scheduling`.
- The daily-run and migration suite then failed because `Database` had no
  `claim_daily_run` API and schema v4 did not exist.
- Independent review identified that a low-level claim could still accept an
  arbitrary caller-supplied date and time. The added boundary test failed
  because a claim one microsecond before 06:50 did not raise.

## Implementation

- Added deterministic Pacific schedule construction for Monday discovery,
  daily preparation, and daily sending.
- Added `not_due`, `due`, and `missed` classification with explicit aware
  instants and kind-specific eligibility windows: 24 hours for discovery,
  10 minutes for preparation, and 5 minutes for sending.
- Added the locked `tzdata` runtime dependency so IANA timezone behavior works
  in the repository's Windows virtual environment as well as on Linux hosts.
- Added schema migration 4 with `daily_runs`, a constrained `claimed` to
  `completed` lifecycle, and a unique opaque-recipient/Pacific-date key.
- Added atomic `BEGIN IMMEDIATE` daily-run claims using SQLite conflict
  handling. Concurrent workers create exactly one row.
- Enforced the Pacific date and 06:50 preparation window inside the same
  transaction that creates the durable claim. Existing claims are checked
  first, so restarts after the window remain idempotent rather than becoming
  errors.
- Added explicit lookup and one-way completion APIs. A restarted scheduler can
  retrieve and complete the original claimed run without creating another.

## Audit hardening on 2026-07-21

An independent real-SQLite probe completed a daily run one microsecond before
its persisted start. Three red cases then required an early completion to fail
without changing the claimed row while permitting completion at the same
instant or later.

`complete_daily_run()` now compares the aware UTC completion instant with the
persisted start inside the write transaction and rejects impossible chronology
before updating state. Focused daily-run, scheduler, and migration verification:
`51 passed`. Ruff and strict mypy were clean.

### Explicit deferrals and decisions

- T05 completion records orchestration completion, not confirmed WhatsApp
  delivery. Enforcement and reconciliation of the actual 07:00 submission
  belongs to T16 after the T15 sender exists.
- The owner approved Monday 00:00 weekly discovery with a 24-hour grace,
  06:50-07:00 preparation, and 07:00-07:05 sending on 2026-07-21. T16 must
  keep any certain retry or ambiguity reconciliation inside the send window;
  it must not introduce next-day catch-up.

one-minute cutoff. The scheduler now selects its cutoff by schedule kind.

eligible microsecond, the exclusive cutoff, and one microsecond after it for
all three schedule kinds. Daily-run claims separately prove that 06:50 and the
last instant before 07:00 are accepted while 07:00 is rejected.
