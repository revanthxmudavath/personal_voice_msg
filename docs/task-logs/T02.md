# T02 — Typed configuration and secret boundaries

## Scope

- Immutable typed settings for development, staging, and production.
- TOML containing only non-secret settings and bounded secret-file names.
- Exact recipient schema with runtime-profile binding.
- Non-leaking sensitive-value wrappers and centralized logging redaction.
- No command-line or repository storage for secret values.

Production secret files are provisioned root-owned outside the repository. The
cloud/container ownership and permission enforcement is part of T18.

## Audit hardening on 2026-07-21

Four audit regressions initially failed:

- staging and production accepted a secret root inside the configuration
  directory;
- `dataclasses.asdict()` exposed `SensitiveValue` internals; and
- no central helper installed redaction on every existing logger handler.

Deployed profiles now require the resolved secret root to be outside the
configuration directory. Development retains a bounded relative-root option
for local testing. `SensitiveValue` is no longer a dataclass, so generic
dataclass serialization fails instead of returning plaintext. The centralized
`install_redacting_filter()` helper installs exactly one redacting filter on
each existing handler and covers normal messages and exception tracebacks.

The focused T01/T02 suite finished with `60 passed`; Ruff and strict mypy were
clean.

### Explicit deferrals

- Root ownership and Unix permission enforcement depend on the deployed
  container/service identity and remain a T18 acceptance gate.
- Each runtime entrypoint must call `install_redacting_filter()` after creating
  its handlers. There is no application entrypoint yet; the first one must add
  an integration test proving this wiring.
- Provider-specific WAHA token-content detection is deferred until T15 defines
  the credential format. Known sensitive filenames and private-key content are
  already rejected by T01 policy.

## Second audit hardening on 2026-07-21

The deployed secret-root check previously used only the configuration file's
immediate directory. A nested `config/settings.toml` could therefore select a
different directory inside the repository. Deployed profiles now discover the
nearest `pyproject.toml` or `.git` boundary and require the resolved secret root
to be outside that entire project. Development behavior is unchanged.

Central redaction now follows logger propagation through parent/root handlers,
deduplicates shared handlers, and replaces an existing `RedactingFilter` with
the newly supplied redactor. This prevents both unfiltered propagated logs and
stale filters after settings are reloaded. Root ownership and Unix mode checks
remain deployment-specific T18 gates and are now named in that task's plan.

The focused T01/T02 suite passed `71` tests and the complete fast suite passed
`209`; Ruff and strict mypy were clean.
