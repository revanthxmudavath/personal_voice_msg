# T04 — Normalization, history search, and deduplication

## Scope and assumptions

- T03 was clean and pushed before T04 began.
- Python 3.12's bundled SQLite 3.45.3 has FTS5 enabled.
- `messages.text` remains the only stored sentence copy.
- Source text is a transient comparison input and is never persisted or logged.
- A conservative false positive is safer than a known duplicate false negative.

## Implementation

- Added NFKC normalization, case folding, punctuation-to-space conversion,
  whitespace collapsing, and SHA-256 normalized hashes.
- Added deterministic six-consecutive-word source comparison.
- Added schema migration 2 with `message_history(message_id, normalized_hash)`.
- Added an external-content FTS5 index over `messages.text`, synchronization
  triggers, and a rebuild during migration for existing version-1 messages.
- Added `MessageHistory` exact, source-copy, and complete-history near-duplicate
  decisions. At one stored sentence per day, complete scoring prevents an FTS
  prefilter or result cap from creating false negatives.
- Removed invisible Unicode format characters during comparison and made stored
  message text immutable so the normalized hash cannot become stale.
- Added RapidFuzz 3.14.5 through the committed lockfile.
- Message insertion and normalized-history insertion share one transaction.

## Corpus and threshold

The expanded corpus contains 14 known duplicate/paraphrase/obfuscation pairs,
12 expected-distinct pairs, and 2 explicitly accepted conservative false
positives caused by word-order insensitivity. Duplicate scores range from
`79.4` to `100.0`; expected-distinct scores range from `35.3` to `58.5`. The
fixed rejection threshold is therefore `79.0`.

No known corpus duplicate is accepted. The documented acceptable false positive
is a genuinely different sentence that reuses nearly the same vocabulary in a
different order. Threshold changes require expanding the corpus and preserving
zero known false negatives.

## Audit hardening on 2026-07-21

A cross-task audit reproduced three T04 failures:

- the public `Database.create_message()` surface could store a near duplicate
  without calling `MessageHistory.evaluate_and_record()`; and
- visible in-word punctuation or symbols (`kind-ness`, `kind💖ness`) could hide
  a copied six-word source span.

The raw public message-insertion methods were removed. Production and ordinary
test setup now enter message history through the atomic evaluate-and-record
path. The one database-constraint regression that must create an invalid row
uses explicit test-local SQL inside a real transaction.

Source-span comparison retains the exact normalized word check and now adds a
conservative letters/numbers/marks compact comparison. This removes visible
punctuation, symbols, and spacing as source-copy evasions without changing the
documented `79.0` history threshold.

Focused migration, delivery-state, history, concurrency, normalization, and
corpus verification: `86 passed`. Ruff and strict mypy were clean.

A second independent review found that the compact comparison was skipped when
removing every separator collapsed six copied source words into fewer than six
candidate tokens. A red regression reproduced the bypass with
`One two three four five six` becoming `Onetwothreefourfivesix`. The premature
candidate token-count return was removed; the exact word path remains bounded,
and the compact comparison now runs whenever the source contains a full span.
The normalization regression file passed all `16` cases, the combined T04/T05
focused suite passed `126` tests, and the complete fast suite passed `198`
tests with the repository-local uv cache configured.
The subsequent T01/T02 audit additions brought the final foundation regression
to `209 passed` without changing T04 behavior.

T10 must still run its planned source-similarity check after generation; the
T04 six-word rule is a deterministic floor, not a complete semantic plagiarism
detector.
